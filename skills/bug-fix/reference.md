# bug-fix reference

## Step 2 信息槽位

> 使用方式：这是查漏表，不是逐条必问题卷。先从用户描述、堆栈、日志、当前代码、最近改动、接口 URL、页面/按钮名、字段名、错误码中预填；只问会影响复现、严重度、定位方向或修复范围的阻塞项。没有证据的根因只能写成"推断，待验证"，并只能输出诊断计划。

| 槽位 | 可从哪里推断 | 何时必须问 |
|------|--------------|------------|
| 错误信息 / 堆栈 | 用户粘贴内容、日志文件、控制台输出、后端异常、前端报错 | 没有任何错误线索且无法定位入口时 |
| 严重度 | 用户描述的影响范围、是否主流程、是否有绕行、是否数据错误 | 严重度会影响处理优先级或看板展示时 |
| 复现步骤 | 用户操作描述、接口路径、页面路由、测试日志、最近变更 | 无法复现且需要区分前端/后端/数据问题时 |
| 触发条件 | 堆栈入参、状态字段、字典值、权限、特定数据、边界值 | 条件决定修复范围或回归范围时 |
| 预期 vs 实际 | 需求文档、接口契约、历史逻辑、用户描述、测试断言 | 用户预期与现有代码/文档冲突时 |
| 服务/模块归属 | 路径、包名、类名、Controller、logger、最近 diff、看板已有记录 | 归属影响修复责任或看板分类时 |

严重度参考：

- **P0 致命**：系统不可用 / 数据丢失 / 主流程完全中断
- **P1 严重**：核心功能不可用，无绕行方案
- **P2 一般**：功能受损但有绕行方案，部分用户受影响
- **P3 轻微**：体验问题 / 非核心功能 / 极少触发

## html-追加格式

追加到 `project-html/data/changes.js` 的 `changes` 数组时的 JS 对象格式（空值字段可省略）：

```js
  {
    kind: "bug",
    service: "<service>",
    module: "<module>",
    title: "<title>",
    date: "<date>",
    severity: "<severity>",
    status: "未修复",
    branch: "<branch>",
    docPath: "<docPath>",
    symptom: "<symptom>",
    stackTrace: `<stackTrace>`,
    reproSteps: [<reproSteps>],
    trigger: "<trigger>",
    expected: "<expected>",
    actual: "<actual>",
    impact: "<impact>",
    codeLocation: "<codeLocation>",
    rootCause: "<rootCause>",
    fixPlan: "<fixPlan>",
    changeList: [<changeList>],
    verifySteps: [<verifySteps>],
    todos: [<todos>],
    assumptions: ["<低风险假设或推断>"],
    conflicts: [
      { point: "<冲突点>", user: "<用户说法>", evidence: "<现有证据>", risk: "<风险>", suggestion: "<建议口径>", blocking: true }
    ],
    blockers: ["<必须先确认/补证据的问题>"],
    openQuestions: ["<非阻塞待确认问题>"]
  },
  // ─── 在此行上方追加新记录 ───
```

**注意**：`stackTrace` 含换行时用模板字面量（反引号）；含反引号时改用双引号并将换行转为 `\n`。其余字符串字段含双引号 → `\"`，含换行 → `\n`。

## 完成后输出格式

```
✅ Bug 文档已生成：docs/bugs/{date}/{task}.md
🐛 HTML 看板已更新：project-html/data/changes.js（浏览器打开 project-html/index.html 查看）
🧭 工作流阶段：Bug Plan Gate 已完成；根因确认后进入 Implementation Gate

📋 关键信息
- 严重度：{severity}
- 初步定位：{codeLocation 摘要，或"待分析"}
- 修复方向：{fixPlan 摘要}

🚀 下一步
1. 打开文档，根据代码定位结果补全「三、根因分析」
2. 若存在 blocker 或根因仍未证实，先按诊断 Todo 补证据；确认后再按「五、代码变更清单」修复
3. 执行「六、验证步骤」确认修复效果
   - 若任一步没有可观察结果，先补充验证口径再执行
4. 修复验证后更新看板状态（浏览器点状态标签只存本地；要全员可见，对 Claude 说：
   "把 project-html/data/changes.js 中标题为「{task}」的记录 status 改为 \"已修复\"，改完跑 node --check"）
5. Review Gate：运行 `/review-fix docs/bugs/{date}/{task}.md` 生成审查任务包，证据包包含 Bug 文档、diff/status、复现证据和验证结果
6. 分发 `/review-check <review-task路径>`，把 findings 贴回 `/review-fix` 汇总修复交接；Critical / Important 处理后重跑验证

【Skill 反馈给 Codex】
- skill：bug-fix
- 本次场景：<一句话描述 Bug 输入形态，如有/无堆栈、有/无复现>
- 运行评价：<顺畅 / 有小问题 / 有阻塞>
- 建议：
  1. <无，或一条可落地的 skill 改进建议>
- 证据：
  - <本次多问/漏问/误判/模板不足的具体表现；没有则写 无>
```
