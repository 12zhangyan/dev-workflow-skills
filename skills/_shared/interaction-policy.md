# 交互与需求判断公共协议

本协议供文档、业务流、Bug、代码阅读和审查类 skill 引用。目标是减少无效提问，避免把猜测写成事实，并把业务逻辑冲突显式暴露出来。

## 四步流程

1. **证据预填**：先读用户输入、当前路径、已有文档、代码入口、Controller/Service/Mapper、枚举/字典、配置、diff、日志或堆栈。能从证据确定的字段直接填写，并在文档里写明依据。
2. **风险分级**：把缺失信息分成低风险、高风险、冲突三类；不要把所有未知都扔给用户。
3. **单点确认**：只问会改变执行方向的问题。每次只问一个，说明为什么这个答案会影响方案、修复或审查结论。
4. **冲突显式记录**：当用户说法与现有代码、字典值、状态机、权限、数据归属、接口契约或表复用约束冲突时，写入冲突记录；不能静默折中，也不能按猜测继续执行。

## 证据引用格式

标注来源时优先给**可定位的引用**，让方案和结论可被复核：
- 代码证据写 `路径:行号` 或 `类名.方法名`（如 `OrderService.java:88`、`OrderService.createOrder()`），不要只写"代码里有"。
- 文档/接口证据写文档路径 + 章节或接口 `method + url`。
- 无法给出精确位置的推断，标为假设或 `待补充`，不要写成既定事实。
- 引用是为了可复核，不是把整段源码抄进文档；给位置和一句结论即可。

## 风险分级

| 类别 | 处理方式 | 例子 |
|------|----------|------|
| 已有证据 | 直接填入，标注来源 `路径:行号`/`类.方法` | 从 Controller 找到 URL；从枚举找到状态值；从 diff 找到改动文件 |
| 低风险未知 | 写明确假设并继续 | 展示文案、非核心命名、模块归属可从路径大致判断 |
| 高风险未知 | 暂停，只问一个阻塞问题 | 状态流转、权限边界、数据归属、接口契约、DB 结构、回滚语义、是否复用既有表 |
| 需求冲突 | 写冲突记录，给建议口径；阻塞时不得生成执行提示 | 用户要求与现有状态机/字典/权限/数据模型相矛盾 |
| 材料不足 | 输出“材料不足，无法下结论”，列缺失材料 | patch 缺新增文件内容；review 没有 diff/文档/入口；Bug 无复现也无定位线索 |

## 冲突记录模板

| 冲突点 | 用户说法 | 现有证据 | 风险 | 建议口径 | 是否阻塞 |
|--------|----------|----------|------|----------|----------|
| <一句话> | <原始需求/描述> | <文件/枚举/接口/日志证据> | <可能影响> | <建议采用的解释或确认问题> | 是/否 |

## 通用输出字段

看板和文档可统一使用这些字段承接判断过程：

- `assumptions`：明确假设，低风险未知可放这里。
- `conflicts`：需求、代码、字典、状态机、权限或数据模型冲突。
- `blockers`：不确认就不能继续编码、修复或审查的问题。
- `openQuestions`：非阻塞待确认项，不影响当前方案继续推进。

## 停止规则

- 存在 `blockers` 或阻塞型 `conflicts` 时，不生成可直接执行的编码/修复操作码；先输出确认问题。
- 根因没有证据时，Bug 文档只能给定位计划，不能给可执行修复 Todo。
- Review 材料不足时，不能输出“未发现问题”；必须输出“材料不足，无法下结论”。

## 交互能力与非交互运行

- 先判断当前会话是否真的能等待用户回复，不按 Codex、Claude Code、Cursor 等产品名猜测能力。
- 交互会话有兼容的结构化提问工具时可使用；没有或调用失败时，立即降级为普通聊天单问，不在同一问题上反复试工具名。
- 非交互/无人值守运行中（含批处理、自动化），不发起等待式提问。低风险未知写入 `assumptions` 后继续；高风险未知或冲突转为 `Blocked`，保留原 finding/任务 ID 并停止对应写入或修复。
- 推荐项不是授权。文件覆盖、接口契约、状态、权限、数据归属、DB、回滚、数据修复等高风险选择，不得因超时、工具不可用或无人回复而采用默认值。
- 输入为空或关键材料缺失时，非交互审查输出 `InsufficientMaterial`；文档/修复类 skill 输出 blocker。都必须列出最小缺失材料和受影响范围，不得卡住等待。

## Skill 维护反馈块

每次 skill 完成后，在最终输出末尾追加一个可复制反馈块。这个块写给维护 skill 的任一智能体，不写给业务执行者；目的是把本次运行中暴露出的规则缺口沉淀下来，方便用户直接复制回来做下一轮优化。

使用规则：

- 如果本次执行顺畅，没有明显规则问题，写 `无`，不要硬凑建议。
- 只记录 skill 层面的改进建议，不记录业务任务本身的下一步。
- 优先记录：多问了不该问的问题、应该问但没问的阻塞问题、误猜需求、冲突没暴露、输出模板缺字段、看板字段不够用、验证步骤不够确定。
- 建议要带触发场景和期望行为，便于直接改 SKILL.md / reference.md。

统一格式：

```text
【Skill 维护反馈】
- skill：<skill-name>
- 本次场景：<一句话描述任务类型/输入形态>
- 运行评价：<顺畅 / 有小问题 / 有阻塞>
- 建议：
  1. <无，或一条可落地的 skill 改进建议>
- 证据：
  - <本次多问/漏问/误判/模板不足的具体表现；没有则写 无>
```
