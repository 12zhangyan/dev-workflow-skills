# 对话移交文档

把当前对话压缩成另一段对话可直接接手的文档。目标是让接手方知道已经做了什么、依据是什么、哪些没有验证，以及下一步如何继续，而不把整段聊天记录或秘密复制过去。

## 共享协议

先遵循 [../_shared/interaction-policy.md](../_shared/interaction-policy.md)。同时读取 [../_shared/workflow-brief.md](../_shared/workflow-brief.md)：已有 `Workflow Brief` 时把它作为索引，不把它误当作完整证据。

非交互/无人值守运行中不等待提问：移交边界不清、关键证据不足、目标路径冲突、写入权限不足或存在会影响接手动作的高风险确认项时，输出 `Blocked` / `NeedsConfirmation` 和最小补充项；不要猜测范围，不覆盖文件，不把推断写成已证实事实。

本 skill 不替代以下任务：

- 新功能或改造方案 → `dev-doc`
- Bug 根因和修复范围 → `bug-fix`
- 多 AI Review 任务或 Review 修复交接 → `review-fix`
- 只读代码审查 → `review-check`
- 已有 findings 的直接修复 → `review-repair`

## 执行流程

### Step 1：确定移交边界

从当前对话提取任务目标、用户明确授权、已经执行的动作、已生成产物、命令及结果、阻塞项和未决问题。只使用本轮确实出现的材料；不要根据“通常应该完成”补写事实。

默认目标是另一段 AI 对话；若用户指定读者（同事、测试、人工）或明确要求格式，按要求调整措辞。没有标题时，根据当前任务生成短横线文件名。

将每条信息标为：

- `已证实`：有当前对话中的文件内容、命令输出、用户确认或可复核路径支撑。
- `推断`：由证据合理推出，但尚未直接验证；说明推断依据。
- `待确认`：影响实现、权限、数据、接口或下一步范围，且当前材料不足。

### Step 2：补充最小可复核证据

对移交文档要引用的本地路径、产物和当前工作区状态，可做最小只读核对；只读取当前任务相关文件，必要时运行项目既有的只读 status 命令。不要为“写得更完整”而扫描无关模块、运行耗时构建或修改代码。

命令结果仅能写实际看到的状态：未运行、失败、`environment-blocked`、部分通过都必须原样保留原因。不要把历史对话中的“已完成”升级为当前已验证。

### Step 3：处理敏感信息与冲突

不得在文档、复制提示或 `Workflow Brief` 中转录 API key、密码、token、cookie、私钥或完整连接串。可记录文件路径、配置键、命中规则和“已脱敏”；若秘密已出现在版本控制历史或对话证据中，提醒接手方按授权轮换，而不是回显其值。

若用户要求与当前证据冲突，保留“待确认/冲突”并引用双方来源；不要静默选边。数据库操作保持只读，结构或数据变更只写待审批建议，不写执行指令。

### Step 4：生成并保存

先读取同一 skill 目录下的完整模板：[reference.md](reference.md)。默认写入当前项目：

首次生成，或“已证实/推断/待确认”与无人值守停机边界容易混淆时，只按当前场景读取一个最小示例：[证据分层示例](examples.md#示例-1partiallycomplete-证据分层移交) / [Blocked 分支](examples.md#示例-2无人值守-blocked-分支)。示例中的路径和结论均为虚构格式，不得复制为当前事实。

```bash
d=$(date +%F) && mkdir -p "docs/handoffs/$d"
```

目标为 `docs/handoffs/<日期>/<任务名>-handoff.md`。写入前检查目标：不存在才创建；已存在且可读时，在交互会话中询问覆盖、时间戳后缀或取消；不可读、状态未知，或无人值守遇到同名文件时输出 `Blocked`，不覆盖。

若当前目录不是用户项目、用户只要求聊天内文本，或没有写入权限，直接输出完整 Markdown 和建议保存路径，不假称文件已写入。

不要登记 HTML 看板：该文档是一次会话的临时交接索引，不是项目知识资产。

### Step 5：交付接手入口

输出已保存路径（或未保存原因）、当前移交状态和一个可复制提示。提示要求新对话先读移交文档，再按“最小读取顺序”打开最多 5 个文件，并明确禁止凭摘要猜测业务、权限、接口或数据库语义。

完成输出必须包含模板中的 `【Workflow Brief】`，`stage` 写当前实际阶段；没有后续动作时 `next`/`nextCommand` 写“人工：确认是否结束/归档”。

## 检查清单

- [ ] 结论均标明已证实、推断或待确认，未把推断写成事实
- [ ] 已完成动作、变更、验证和 blocker 都附最小证据或明确“未验证”
- [ ] 不含秘密值，也没有要求接手方执行未获授权的写操作
- [ ] 文件冲突已按 Step 4 处理，未覆盖已有移交文档
- [ ] 接手提示与最小读取顺序完整，首轮不超过 5 个文件
- [ ] 最终输出包含 `【Workflow Brief】`
